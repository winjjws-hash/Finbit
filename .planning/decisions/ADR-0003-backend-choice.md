# ADR-0003. 백엔드와 저장 방식 선택

## 상태

Accepted

## 배경

Energy Budget은 개인이 자신의 컨디션과 할 일을 기록하는 앱이다. 초기 MVP에서는 사용자 계정, 친구 공유, 실시간 동기화가 필수 기능이 아니다. 가장 중요한 것은 사용자가 빠르게 입력하고, 앱이 에너지 예산과 추천 결과를 보여주는 것이다.

백엔드 선택 기준은 다음과 같다.

- 6주 안에 혼자 구현 가능해야 한다.
- 첫 실행과 데모가 간단해야 한다.
- 네트워크 연결 없이도 기본 흐름을 보여줄 수 있어야 한다.
- 이후 필요하면 서버나 클라우드 저장소로 확장할 수 있어야 한다.

## 비교 대상

| 선택지 | 장점 | 단점 | 적합도 |
|---|---|---|---|
| 로컬 저장소 | 구현이 쉽고 오프라인 사용 가능 | 기기 간 동기화 불가 | 매우 좋음 |
| Firebase | 인증, DB, 배포 생태계가 좋음 | 설정과 보안 규칙 학습 필요 | 보통 |
| REST API 서버 | 구조가 명확하고 확장성 좋음 | 백엔드 개발 시간이 추가됨 | 낮음 |
| Supabase | DB와 인증을 빠르게 구성 가능 | 외부 서비스 의존과 설정 필요 | 보통 |

## 결정

초기 MVP에서는 **백엔드 없이 로컬 저장소 우선 구조**를 사용한다.

## 결정 이유

1. Energy Budget의 Must 기능은 서버 없이도 충분히 검증할 수 있다.
2. 네트워크 연결이 없어도 컨디션 입력, 할 일 등록, 추천 결과 저장 흐름을 데모할 수 있다.
3. 백엔드 구축 시간을 줄이고 모바일 UI와 핵심 추천 규칙에 집중할 수 있다.
4. Data 레이어에 Repository 인터페이스를 두면 이후 Firebase나 REST API로 확장하기 쉽다.

## 결과

- 초기 구현은 `src/data/local`에 로컬 저장 구현을 둔다.
- 데이터 접근 방식은 `src/data/repositories`의 인터페이스를 통해 감싼다.
- MVP 단계에서는 로그인, 서버 API, 클라우드 동기화를 제외한다.
- 실제 영구 저장은 Expo 환경에 맞춰 AsyncStorage로 확장할 수 있다.

## 대안

### Firebase

나중에 사용자 계정과 기기 간 동기화가 필요해지면 좋은 선택이다. 하지만 초기 MVP에서는 설정과 보안 규칙이 추가 부담이 된다.

### REST API 서버

백엔드 학습과 확장성에는 좋지만, 이번 프로젝트 목표는 모바일 앱의 핵심 흐름을 먼저 완성하는 것이므로 제외한다.
