# ADR-0002. 상태관리 방식 선택

## 상태

Accepted

## 이 결정은 무엇인가?

Energy Budget에서 사용자가 입력하는 값들을 앱 안에서 어떻게 관리할지 정하는 문서이다.

여기서 말하는 상태는 어려운 개념이 아니라, 앱 화면에서 계속 바뀌는 값이다.

- 오늘의 피로도
- 오늘의 기분
- 남은 시간
- 오늘 해야 할 일 목록
- 각 할 일의 에너지 소모량
- 완료했는지 여부
- 추천받은 오늘의 계획

쉽게 말해, 사용자가 앱을 쓰면서 바꾸는 값들을 어디서 어떻게 들고 있을지 결정하는 것이다.

## 배경

`01-requirements.md`에 따르면 Energy Budget의 핵심 시나리오는 세 가지이다.

1. 아침에 오늘 감당 가능한 계획을 세운다.
2. 하루 중 계획이 무너졌을 때 다시 조정한다.
3. 일주일 기록을 보고 생활 패턴을 돌아본다.

이 세 시나리오에서 공통으로 필요한 것은 거창한 서버 상태가 아니라, 사용자가 방금 입력한 값을 화면에서 바로 계산하고 보여주는 흐름이다.

예를 들어 사용자가 피로도를 4에서 8로 바꾸면, 앱은 바로 "오늘 에너지가 부족할 수 있다"는 식으로 추천 결과를 바꿔야 한다. 그래서 상태관리는 이 앱에서 꽤 중요하지만, 처음부터 복잡한 도구를 쓸 필요는 없다.

## 결정

초기 MVP에서는 **React 기본 상태 + ViewModel 함수 구조**를 사용한다.

즉, 화면에서 바뀌는 값은 React의 기본 상태로 관리하고, 화면에 보여줄 계산 결과는 ViewModel 함수에서 정리한다.

## 왜 이 방식을 선택했나?

### 1. 지금 필요한 기능에 충분하다

Energy Budget의 Must 기능은 다음 다섯 가지이다.

- 오늘의 컨디션 입력
- 할 일 등록
- 에너지 예산 계산
- 오늘 할 일 추천
- 일일 계획 결과 저장

이 기능들은 처음에는 한 사람의 하루 기록만 다루기 때문에 복잡한 전역 상태관리 도구가 없어도 만들 수 있다.

### 2. 초급자가 이해하기 쉽다

처음부터 Redux 같은 도구를 쓰면 앱보다 상태관리 도구를 이해하는 데 시간이 더 많이 걸릴 수 있다.

이번 프로젝트에서는 "피로도 값이 바뀌면 추천 결과도 바뀐다" 정도의 흐름을 직접 이해하는 것이 더 중요하다.

### 3. 발표할 때 설명하기 쉽다

이 방식은 다음처럼 설명할 수 있다.

> 사용자가 입력한 값은 화면 상태로 들고 있고, 에너지 계산은 별도 함수로 분리했습니다. 그래서 화면은 입력과 표시를 담당하고, 계산 규칙은 따로 관리할 수 있습니다.

### 4. 나중에 확장할 수 있다

초기에는 React 기본 상태로 시작하지만, 화면이 많아지고 여러 화면이 같은 데이터를 공유해야 하면 Zustand 같은 가벼운 상태관리 도구로 옮길 수 있다.

## 비교한 선택지

| 선택지 | 쉬운 설명 | 장점 | 단점 | 이번 선택 |
|---|---|---|---|---|
| React 기본 상태 | 화면 안에서 값을 관리하는 기본 방식 | 쉽고 빠르다 | 앱이 커지면 복잡해질 수 있다 | 선택 |
| ViewModel 함수 | 화면에 보여줄 값을 미리 정리하는 함수 | 화면 코드가 깔끔해진다 | 구조를 지켜야 효과가 있다 | 선택 |
| Zustand | 작은 전역 저장소 | 가볍고 실용적이다 | 지금은 없어도 된다 | 나중 후보 |
| Redux Toolkit | 큰 앱용 상태관리 도구 | 규칙이 명확하다 | 초반 설정과 개념이 무겁다 | 제외 |
| React Query | 서버 데이터 관리 도구 | API 데이터에 강하다 | 현재 MVP는 서버가 없다 | 제외 |

## 적용 방식

Energy Budget에서는 상태와 계산을 다음처럼 나눈다.

| 필요한 값/기능 | 둘 위치 | 이유 |
|---|---|---|
| 사용자가 입력한 피로도, 기분, 남은 시간 | `presentation/screens` | 화면에서 바로 바뀌는 값이기 때문 |
| 오늘 할 일 목록 | `presentation/screens` 또는 `application/state` | 화면에서 추가/수정되는 값이기 때문 |
| 화면에 보여줄 에너지 예산, 상태 문구 | `application/view_models` | 화면에 맞게 정리된 결과이기 때문 |
| 에너지 예산 계산식 | `domain/rules` | 앱의 핵심 규칙이기 때문 |
| 완료 여부와 일일 기록 저장 | `data` | 기록을 저장하고 불러오는 책임이기 때문 |

## 결과

- MVP에서는 Redux Toolkit을 사용하지 않는다.
- MVP에서는 Zustand도 바로 사용하지 않는다.
- 화면 상태는 React 기본 상태로 시작한다.
- 에너지 계산 규칙은 `src/domain/rules`에 둔다.
- 화면에 보여줄 데이터 정리는 `src/application/view_models`에 둔다.
- 이후 주간 리포트나 여러 화면 공유가 복잡해지면 Zustand 도입을 다시 검토한다.

## 발표용 60초 설명

Energy Budget은 사용자가 피로도, 기분, 남은 시간, 할 일을 입력하면 그 값에 따라 오늘의 에너지 예산과 추천 결과가 바뀌는 앱입니다. 그래서 상태관리가 필요하지만, 초기 MVP에서는 화면 수가 많지 않고 서버 데이터도 없기 때문에 Redux 같은 무거운 도구는 사용하지 않기로 했습니다. 대신 React 기본 상태로 입력값을 관리하고, 에너지 계산 규칙은 `domain/rules`에 따로 분리했습니다. 이렇게 하면 초급자도 흐름을 이해하기 쉽고, 나중에 앱이 커졌을 때 Zustand 같은 도구로 확장할 수 있습니다.

## 내가 이해했는지 확인하는 질문

- 피로도와 기분처럼 사용자가 바꾸는 값은 왜 상태인가?
- 에너지 계산식을 화면 코드 안에 길게 쓰지 않는 이유는 무엇인가?
- Redux Toolkit을 이번 MVP에서 제외한 이유는 무엇인가?
- 화면이 많아지고 데이터 공유가 복잡해지면 어떤 도구를 다시 검토할 수 있는가?
