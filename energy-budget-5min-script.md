# Energy Budget 5분 발표 대본

## 발표 전 준비

1. PC에서 GitHub Pages 또는 최종 평가 PDF를 먼저 열어둔다.
2. 스마트폰 알람 또는 타이머를 **5분**으로 맞춘다.
3. 화면이 준비되면 타이머를 시작하고 발표한다.
4. 발표 구성은 **4분 30초 설명 + 30초 시연 데모**로 진행한다.

발표 URL:

- GitHub Pages: https://winjjws-hash.github.io/Finbit/
- 최종 평가 PDF: https://winjjws-hash.github.io/Finbit/Energy_Budget_Final_Evaluation_Presentation.pdf

## 1쪽 표지 - 15초

안녕하세요. 제 프로젝트는 **Energy Budget**입니다.  
저는 **2024136064 정원식**입니다.  

Energy Budget은 **오늘 내 컨디션과 남은 에너지를 먼저 계산해서, 지금 할 일과 미뤄도 되는 일을 나눠주는 계획 앱**입니다.

쉽게 말하면, 할 일을 무조건 많이 하게 만드는 앱이 아니라 **오늘 내 상태에 맞춰 할 일을 다시 정리해주는 앱**입니다.

## 2쪽 평가 기준 대응표 - 15초

이번 발표는 교수님 평가 기준에 맞춰 준비했습니다.  
비전, 문제 정의, WBS, 기술 스택, 구현 방법, 테스트, 배포, Q&A까지 모두 하나의 흐름으로 설명하겠습니다.

발표는 5분 타이머를 켜고 진행하며, 마지막 30초는 실제 앱 시연으로 보여드리겠습니다.

## 3쪽 비전 - 25초

Energy Budget의 비전은 하루 계획을 **시간 중심에서 에너지 중심으로 바꾸는 것**입니다.

기존 계획 앱은 할 일을 얼마나 많이 넣을지에 집중합니다.  
하지만 실제 생활에서는 시간이 남아도 피곤하면 계획을 지키기 어렵습니다.

그래서 저는 더 많이 하게 만드는 앱보다, 오래 지속할 수 있는 하루를 만드는 앱을 목표로 했습니다.

## 4쪽 문제 정의 - 25초

제가 정의한 문제는 계획 실패를 단순히 의지 부족으로 보는 것입니다.

사람들은 시간이 없어서만 계획을 실패하지 않습니다.  
시간은 남아도 피로하거나 기분이 좋지 않으면 집중력과 실행력이 떨어집니다.

그래서 이 프로젝트는 계획 실패를 개인의 문제가 아니라 **컨디션을 고려하지 않은 과부하 설계 문제**로 봤습니다.

## 5쪽 제품 흐름 - 25초

앱의 흐름은 간단합니다.

먼저 사용자가 피로도, 기분, 남은 시간, 할 일을 입력합니다.  
그다음 앱이 오늘 감당 가능한 에너지 예산을 계산합니다.  
마지막으로 할 일을 **지금 하면 좋은 일**과 **나중으로 미뤄도 되는 일**로 나눕니다.

또 마감 3시간 전과 1시간 전에는 알림 현황에서 먼저 확인할 수 있게 했습니다.

## 6쪽 시연 데모 준비 - 15초

시연은 사용자 시나리오 순서로 보여드리겠습니다.

닉네임 입력, 컨디션 조절, 할 일 추가, 자동 추천, 달력과 알림 확인 순서입니다.

핵심은 할 일을 더 넣는 것이 아니라, 오늘 무리하지 않아도 되는 기준을 보여주는 것입니다.

## 7쪽 WBS 진행 현황 - 25초

WBS는 10주차부터 15주차까지 나누어 진행했습니다.

10주차에는 비전, 문제 정의, 요구사항, WBS를 완료했습니다.  
11주차에는 ADR과 아키텍처, 개발 환경을 정리했습니다.  
12주차와 13주차에는 핵심 입력 기능과 추천 기능을 구현했습니다.

14주차에는 날짜별 저장, 월간 달력, 알림 현황을 확장했고, 15주차에는 테스트, README, AGENTS.md, 최종 PDF까지 정리했습니다.  
그래서 현재 발표 기준 산출물은 모두 완료 상태입니다.

## 8쪽 FM WBS - 20초

이 WBS는 산출물 중심으로 정리했습니다.

기획, 설계, 핵심 입력, 추천 로직, 기록과 알림, 검증과 발표까지 모두 완료로 관리했습니다.  
Owner는 모두 정원식으로 두었고, 각 항목마다 완료 기준을 붙였습니다.

## 9쪽 기술 스택 - 25초

기술 스택은 React Native, Expo, TypeScript를 사용했습니다.

React Native는 모바일 앱 UI를 만들기 위해 사용했고, Expo는 개발 서버와 웹 미리보기를 빠르게 실행하기 위해 사용했습니다.  
TypeScript는 입력값과 계산 로직의 오류를 줄이기 위해 사용했습니다.

저장은 현재 AsyncStorage를 사용했고, 발표 자료와 WBS는 GitHub Pages로 바로 확인할 수 있게 했습니다.

## 10쪽 앱 구조와 아키텍처 - 25초

앱 구조는 네 개의 레이어로 나누었습니다.

사용자가 보는 화면은 presentation, 화면 흐름은 application, 에너지 계산 규칙은 domain, 저장 기능은 data에 두었습니다.

이렇게 나눈 이유는 화면 코드와 계산 규칙이 섞이지 않게 하기 위해서입니다.  
그래야 나중에 기능을 수정하거나 발표에서 구조를 설명하기 쉽습니다.

## 11쪽 구현 방법 - 30초

구현은 감정적인 문제를 계산 가능한 흐름으로 바꾸는 방식으로 했습니다.

피로도, 기분, 남은 시간을 입력하면 `calculateEnergyBudget`이 오늘의 에너지 예산을 계산합니다.  
그리고 각 할 일의 에너지 소모량을 비교해서 예산 안에 들어오면 지금 할 일로 추천하고, 넘치면 미루기 후보로 분리합니다.

할 일은 추가한 뒤에도 제목, 날짜, 마감 시간, 카테고리, 에너지 소모를 수정할 수 있습니다.

## 12쪽 ADR 요약 - 25초

ADR은 총 4개를 준비했습니다.

첫 번째는 모바일 플랫폼으로 React Native와 Expo를 선택한 이유입니다.  
두 번째는 상태관리 방식으로, 초기 MVP에서는 React 기본 상태와 도메인 분리를 선택했습니다.  
세 번째는 저장 방식으로, 현재는 서버 없이 로컬 저장으로 핵심 가치를 검증했습니다.  
네 번째는 Energy Budget이라는 주제를 선택한 이유입니다.

질문이 나오면 ADR의 상황, 결정, 이유, 결과 순서로 답변하겠습니다.

## 13쪽 개발 환경, 빌드, 배포 - 25초

개발 환경은 `npm install`로 의존성을 설치하고, `npm run start` 또는 `npm run web`으로 Expo 개발 서버를 실행합니다.

검증은 `npm run typecheck`와 `npm run test`로 확인했습니다.

빌드는 소스코드를 실행 가능한 산출물로 만드는 과정이고, 배포는 그 산출물을 다른 사람이 접근할 수 있는 위치에 올리는 과정입니다.  
발표 자료와 WBS는 GitHub Pages에서 URL로 확인할 수 있게 준비했습니다.

## 14쪽 개발자 기본소양 - 30초

개발하면서 가장 큰 시행착오는 기능을 너무 많이 넣으려 했던 점입니다.

처음에는 AI 추천, 리포트, DB 저장까지 생각했지만, WBS와 MoSCoW로 범위를 줄여 MVP 핵심 기능에 집중했습니다.

성능 면에서는 필요한 계산만 다시 수행하도록 구성했고, 모바일 화면에서 카드와 여백을 줄여 사용성을 개선했습니다.  
코드 품질은 TypeScript 타입, 도메인 규칙 분리, Repository 구조로 관리했습니다.

테스트는 단위 관점에서 에너지 계산 규칙을 확인하고, 통합 관점에서 컨디션 입력부터 추천 분리까지의 사용자 흐름을 확인했습니다.

## 15쪽 Q&A 준비 - 25초

질문에 대비해서 ADR 기준으로 답변을 준비했습니다.

DB 질문이 나오면, 현재는 AsyncStorage로 날짜별 계획을 저장하고 있고 SQLite는 다음 확장 단계라고 답할 수 있습니다.

앱 구조 질문에는 presentation, application, domain, data로 나누었다고 설명할 수 있습니다.

Expo를 쓴 이유는 6주 안에 모바일 MVP를 빠르게 만들고 Android와 Web export를 검증하기 위해서입니다.

## 16쪽 마무리 - 15초

정리하면 Energy Budget은 더 많이 하게 만드는 앱이 아니라, 오늘의 나를 기준으로 계획을 다시 세우게 돕는 앱입니다.

앞으로는 SQLite 저장, 주간 리포트, 요일별 에너지 패턴 분석으로 발전시킬 계획입니다.

이제 30초 시연을 보여드리겠습니다.

## 30초 시연 데모 대사

먼저 닉네임을 입력하면 날짜별 계획이 저장됩니다.  
피로도와 남은 시간을 조절하면 오늘의 에너지 예산이 바뀝니다.  
할 일을 추가할 때 예상 시간, 날짜, 마감 시간, 에너지 소모를 입력합니다.  
앱은 이 값을 기준으로 지금 하면 좋은 일과 나중으로 미룰 일을 나눕니다.  
마지막으로 달력에서는 날짜별 계획과 과제가 많은 날을 확인할 수 있고, 알림 현황에서 마감 알림도 확인할 수 있습니다.
