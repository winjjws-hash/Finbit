# Energy Budget - 위험 식별과 대응

## 1. 문서 목적

이 문서는 Energy Budget 프로젝트를 6주 안에 완성하기 위해 예상되는 위험과 대응 방안을 정리한다. Week 10 기준에 맞춰 최소 5개 이상의 위험을 식별한다.

## 2. 주요 위험 요약

| 번호 | 카테고리 | 위험 | 영향 | 대응 방안 |
|---|---|---|---|---|
| R1 | 기술 | React Native와 Expo 환경 구축에서 시간이 오래 걸릴 수 있음 | 초반 개발 지연 | Hello World 실행을 먼저 완료하고, 복잡한 네이티브 기능은 제외한다. |
| R2 | 기능 범위 | 에너지 추천 로직을 너무 복잡하게 만들 수 있음 | MVP 완성도 저하 | 초기에는 피로도, 기분, 남은 시간, 에너지 소모량만 사용한 단순 점수식으로 시작한다. |
| R3 | 일정 | 설계, 구현, 발표 준비를 혼자 모두 해야 함 | 마감 직전 작업 과부하 | Must 기능을 먼저 완성하고 Should/Could 기능은 시간이 남을 때만 진행한다. |
| R4 | 데이터 저장 | 일일 기록 저장 방식이 늦게 정해질 수 있음 | 리포트 기능 구현 지연 | 초기에는 로컬 저장소 우선으로 진행하고 서버 연동은 제외한다. |
| R5 | 사용성 | 사용자가 입력해야 하는 값이 많으면 앱을 자주 쓰기 어려움 | 실생활 사용성 저하 | 입력 항목을 피로도, 기분, 남은 시간, 할 일 정도로 최소화한다. |
| R6 | AI 의존 | 바이브코딩으로 만든 코드를 본인이 이해하지 못할 수 있음 | 발표와 수정 어려움 | ADR, architecture 문서를 작성하고 핵심 계산 로직은 직접 설명할 수 있게 정리한다. |
| R7 | 발표 | 앱의 차별점을 투두앱과 구분해서 설명하지 못할 수 있음 | 발표 설득력 저하 | "시간 관리가 아니라 에너지 관리"라는 핵심 메시지를 반복해서 사용한다. |

## 3. 가장 큰 위험

가장 큰 위험은 **기능 범위가 커져서 MVP를 완성하지 못하는 것**이다.

Energy Budget은 아이디어상 감정 분석, AI 상담, 캘린더 연동, 웨어러블 연동까지 확장할 수 있다. 하지만 이번 프로젝트의 목표는 6주 안에 실행 가능한 MVP를 만드는 것이다. 따라서 초기 버전은 사용자가 직접 입력한 컨디션과 할 일을 바탕으로 에너지 예산을 계산하고, 오늘 할 일을 추천하는 흐름에 집중한다.

## 4. 대응 원칙

1. Must 기능을 먼저 완성한다.
2. 외부 API와 서버 기능은 초기 범위에서 제외한다.
3. 추천 로직은 설명 가능한 단순 규칙으로 시작한다.
4. 매주 실행 가능한 결과물을 남긴다.
5. AI가 만든 코드는 본인이 읽고 설명할 수 있는 수준으로 유지한다.

## 5. 발표용 요약

Energy Budget의 가장 큰 위험은 기능을 너무 많이 넣으려다가 핵심 MVP를 완성하지 못하는 것이다. 그래서 초기에는 외부 캘린더, AI 상담, 웨어러블 연동을 제외하고, 컨디션 입력, 할 일 등록, 에너지 계산, 오늘 할 일 추천, 일일 기록 저장에 집중한다.
